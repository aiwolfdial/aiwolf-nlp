こちらのページでは、大会の **本戦フェーズ** での具体的な手順を説明します。

本戦は **他チームのエージェントと実際に対戦** するフェーズです。運営サーバが対戦のマッチングを担い、参加チームは指定された時刻までにエージェントを起動・待機させておく形で進行します。

> 予選・本戦に共通する `config.yml` の書き方や、トラックの考え方は [大会の進め方（全体像）](./competition.md) を先にお読みください。
> 予選提出がまだの方は [予選編](./qualifying.md) を先に。**予選未提出のままでは本戦参加できません**。

---

## 1. 本戦で何が起きるか

本戦フェーズでは、運営サーバが **複数チームのエージェントをマッチングして対戦** させます。

```text
あなたのエージェント（トラックの人数ぶん起動・待機）
       │
       │ 開始時刻までに接続して待機
       ▼
   本戦サーバ（運営が時刻指定で開始）
       │
       └─→ 他チームのエージェントとマッチング
              │
              ▼
          本戦試合の進行
              │
              ▼
          サーバ側にログ記録
```

重要なのは「**指定時刻までに接続済みでなければ、その対戦に参加できない**」という点です。
予選のように「自分が好きなタイミングで接続して試合を回す」のとは、運用が逆転します。

---

## 2. 開始時刻に関するルール

本戦には **明確な開始時刻** が設定されます。運営から事前に案内される、その時刻を厳守してください。

| 確認事項 | 内容 |
|---|---|
| **予選提出の確認** | 開始時刻までに **予選ログの提出** が確認できないチームは、自動的に **辞退とみなされる** ことがあります |
| **エージェント起動・待機** | 開始時刻までに **エージェントを起動して、本戦サーバへ接続済み** にしておきます |
| **複数トラック参加** | 参加するトラックの数だけエージェントを **同時に** 待ち受けさせます |
| **直前のコード変更** | 本戦開始までは、エージェントのコード・プロンプトを **修正しても問題ありません** |

> **ポイント**：本戦は **遅刻・接続漏れがそのまま不戦敗**につながります。直前にバタバタしないよう、開始時刻の **30分以上前** に起動しておくのが安全です。

---

## 3. 接続のための設定

`config/config.yml` の `web_socket` を、運営から配布される **本戦用** の URL に書き換えます。
**本戦用の URL は予選とは別** なので、上書きせずに **トラック × フェーズ** ごとに `config.yml` を分けて管理するのがおすすめです。

```yaml
web_socket:
  url: ws://（運営配布の本戦用URL）/ws
  token:                # 配布された場合のみ設定
  auto_reconnect: true  # ★必ず true（後述）

agent:
  num: 5                # トラックの人数に合わせる
  team: kanolab         # 参加登録時のチーム名
  kill_on_timeout: true
```

### `auto_reconnect: true` がなぜ重要か

本戦中は、ネットワークやサーバ側の都合で **一時的に接続が切れること** があります。`auto_reconnect: true` でないと、

* 接続失敗時に **再試行されない**
* 一度切断されたあと **次の試合に参加できない**

という状態になり、不戦敗のリスクが上がります。**必ず `true` にしてください**。

---

## 4. 起動と接続確認

開始時刻の少し前にエージェントを起動します。

```bash
cd ~/aiwolfdial/aiwolf-nlp-agent-llm
uv run python src/main.py
```

正常に起動すると、ターミナルに **接続成功・パケット受信のログ** が流れ始めます。
最初のうちは `NAME` や `INITIALIZE` などのリクエストが届くのを目視で確認するとよいです（[ログの読み方](../evaluation/agent_log_guide.md) を参照）。

### 接続できたら運営に一言

運営から **接続確認の連絡を求められる** ことがあります。指定された Slack スレッド等で、「接続できました」と一言返信しておくと、運営側でも全チームの起動状況を把握できます。

### 接続できないとき

エラーが出る場合は、**エラーログの先頭数十行をそのままコピーして** 運営に共有してください。
原因の多くは次のいずれかです。

* `web_socket.url` が **予選用のまま** になっている
* `web_socket.auto_reconnect` が **`false`** のまま
* `agent.num` が **トラック人数と不一致**
* `agent.team` が **参加登録時のチーム名と不一致**
* APIキー切れ・レート制限到達

---

## 5. 本戦中の運用

### 試合の進行はサーバ主導

予選と違い、**自分のタイミングではなく運営のスケジュール** で進みます。
基本的には起動して放置するだけで、開始時刻になればサーバから対戦リクエストが届き始めます。

### トラックごとに順次実行されることがある

参加チーム全体の **API レート制限・課金量への配慮** から、運営側でトラックを **軽い順に1つずつ** 走らせることがあります（例：5人村が完了してから9人村を開始）。

* 自分が参加しているトラックの **順番** を把握しておく
* 自分の出番でないあいだも、**接続を維持** しておく（`auto_reconnect: true` でOK）

### コード変更・再起動は試合の合間に

本戦中にエージェントの不具合に気づいた場合、**試合の合間** であれば停止・修正・再起動が可能です。
試合中に強制停止すると、その試合は不戦敗になるので、**進行中の試合を優先** してください。

### 接続が切れたら

`auto_reconnect: true` ならエージェントが自動で再接続を試みます。
それでも復旧しない場合は、運営に連絡しつつ手動で再起動してください。

---

## 6. 本戦が終わったら

最後の試合が完了すると、サーバ側で本戦終了が宣言されます。

### 結果発表まで時間がかかります

人狼知能大会では、**勝率に加えて人手評価・LLM Judge による主観評価** が行われることが多く、最終結果が出るまで **数日〜数週間** かかります。

* 本戦終了直後 → サーバ側で勝率集計
* その後 → **人手評価**（運営が試合ログを目視で評価）
* 加えて → **LLM Judge** による自動評価（[LLM Judge の使い方](../evaluation/llm_judge_usage.md)）
* それらを総合 → 最終順位の発表

> **ポイント**：勝率トップ＝優勝、ではないことが多いです。
> 「自然な議論ができていたか」「人狼らしいふるまいだったか」など、**主観評価項目** で順位が動きます。
> 評価のされ方は [改良のヒントと課題点](../enhancement/improvement_tips.md) も参照してください。

### 論文・成果物の提出

学会連動の大会では、本戦終了後に **論文・成果報告書の提出** を求められることがあります。
これは運営から別途案内されるので、**Slack 等の連絡を見逃さない** ようにしてください。

## 7. 本戦前のチェックリスト

開始時刻の数時間前から、次の項目を確認しておくとトラブルを最小化できます。

* [ ] 予選提出が完了している
* [ ] **本戦用の `config.yml`**（接続先 URL が本戦用になっている）を用意した
* [ ] 参加するすべてのトラックぶん `config.yml` がある
* [ ] **`auto_reconnect: true`** になっている
* [ ] `agent.num` がトラックの人数と一致
* [ ] `agent.team` が参加登録時のチーム名と一致
* [ ] APIキー残量・レート制限の上限を確認した
* [ ] **開始の30分以上前** にエージェントを起動した
* [ ] 接続成功のログを確認した
* [ ] 運営の指定する **Slack スレッド** で接続報告を済ませた
* [ ] 何かあったときにすぐ連絡できる手段を確保している
