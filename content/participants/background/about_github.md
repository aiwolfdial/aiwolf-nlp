このページは、開発で必ず使う **Git** と **GitHub** を、最低限の流れで押さえます。
用語はその都度かんたんに説明し、ページ末尾に一覧をまとめています。

---

## 目的：なぜ Git を使うのか

ファイルを手作業で管理すると、こんな問題が起きます。

* どれが最新版か分からなくなる
* 古い版で上書きしてしまう
* いつ・誰が・何を変えたのか追えない

**Git** は、ファイルの **変更履歴を記録する仕組み（バージョン管理システム）** です。
「いつ・誰が・何を変えたか」が残り、過去の状態に戻すことも、複数人の変更を後で1つにまとめることもできます。

---

## Git と GitHub の違い

よく混同されますが、役割が違います。

* **Git** … 自分の **PC（手元）** で変更履歴を管理する **仕組み**。
* **GitHub** … その履歴を **インターネット上** に置いて、共有・バックアップ・共同作業できる **サービス（場所）**。

ひとことで言うと、**Git が「仕組み」、GitHub が「置き場所」** です。手元の Git の履歴を GitHub に送れば、他の人と共有したり、PC が壊れても復元したりできます。

> このとき、履歴ごと管理されるプロジェクト一式を **リポジトリ（repository）** と呼びます。手元にあるものを **ローカルリポジトリ**、GitHub 上にあるものを **リモートリポジトリ** と言います。

---

## 使い方①：既存のリポジトリを使う

まずは、すでに GitHub にあるリポジトリを **手元に持ってきて動かす** 流れです。サンプルエージェントを使うときも、これにあたります。

### 1. クローンする

**クローン（clone）** は、GitHub 上のリポジトリを **丸ごと手元に複製** することです。

```bash
git clone https://github.com/<owner>/<repo>.git
cd <repo>
```

これで、リポジトリのファイル一式が履歴つきで手元にコピーされ、そのフォルダに移動できます。

### 2. README を読んで環境構築する

クローンしたら、まず **`README.md`** を読みます。
`README.md` は、GitHub でリポジトリを開いたときに **一覧の下に自動表示される説明書** で、そのプロジェクトの概要・必要環境・セットアップ手順・使い方が書かれています。

多くの場合、README に従って次のような準備をします（内容はリポジトリごとに異なります）。

```bash
# 例：README に書かれた手順に沿って環境を整える
python3 -m venv .venv            # 仮想環境を作る
source .venv/bin/activate        # 仮想環境を有効化
pip install -r requirements.txt  # 必要なパッケージを入れる
```

> README は「他人（数か月後の自分も含む）がゼロから動かせる」ことを目指して書かれています。自分がリポジトリを作る側になったときも、README を整えておくと親切です。

## 使い方②：自分のリポジトリを用意する

自分のプロジェクトを GitHub で管理したいときの流れです。

### GitHub で新規リポジトリを作る

1. GitHub にログインし、右上の **「＋」→「New repository」**
2. リポジトリ名（例：`my-project`）を入力し、Public / Private を選ぶ
3. **Create repository** を押すと、リポジトリの URL が表示される

手元のプロジェクトを、この新しいリポジトリに接続して送ります。

```bash
cd my-project
git init                          # このフォルダで Git を使い始める
git remote add origin https://github.com/<you>/my-project.git  # 送り先を登録
```

> ここで出てきた **リモート（remote）** は「送り先の GitHub リポジトリ」のこと。その既定の呼び名が **`origin`** です。以降 `origin` と書けば、この GitHub リポジトリを指します。

あとは後述の `git add → commit → push` で、変更を GitHub に送れます。

---

## よく使うコマンド

日々の開発は、ほぼこの流れの繰り返しです。

```bash
git status                       # 今どのファイルを変更したか確認
git add .                        # 変更を「次に記録する分」として登録
git commit -m "feat: 新機能を追加"  # 変更を1区切りとして記録
git push origin main             # 記録を GitHub（origin）へ送る
git pull origin main             # GitHub 側の最新を手元に取り込む
```

それぞれの意味は次のとおりです。

| コマンド | 何をする | 用語 |
|---|---|---|
| `git status` | 変更したファイルの状況を表示 | — |
| `git add .` | 変更を **ステージ**（次のコミットに含める仮置き）に登録 | ステージ |
| `git commit -m "..."` | 変更をひと区切りとして記録 | **コミット**＝変更の記録1つ |
| `git push` | 手元の記録を GitHub へ送る | **プッシュ** |
| `git pull` | GitHub の最新を手元へ取り込む | **プル** |
| `git remote -v` | 送り先（origin など）の URL を確認 | — |
| `git remote set-url origin <URL>` | 送り先の URL を変更 | — |

### コミットメッセージの書き方

`-m "..."` に書くメッセージは、後から履歴を読む人（自分含む）のためのものです。先頭に **種類（type）** を付けると、何の変更か一目で分かります。

```bash
git commit -m "feat: 新機能を追加"
git commit -m "fix: ログイン時のバグを修正"
git commit -m "refactor: 処理を整理（動作は変えない）"
```

よく使う種類は次のとおりです。

| type | 使う場面 |
|---|---|
| `feat` | 新しい機能の追加 |
| `fix` | バグの修正 |
| `docs` | ドキュメント（README など）の変更 |
| `refactor` | 動作を変えずにコードを整理・改善 |
| `chore` | 設定・依存更新など、上記以外の雑務的な変更 |

## `.gitignore`：送ってはいけないものを除外する

`git add .` はフォルダ内の変更をまとめて登録しますが、**GitHub に送ってはいけないファイル** もあります。代表が **`.env`**（APIキーなどの秘密情報。→ [環境変数と.envファイル](./about_env_vars.md)）です。これを公開リポジトリに送ると、鍵が漏洩します。

これを防ぐのが **`.gitignore`** です。リポジトリ直下にこのファイルを置き、追跡しないものを書いておくと、Git はそれらを記録対象から外します。

```text
.env            # 秘密情報（最重要）
.venv/          # 仮想環境（手元で作り直せる）
__pycache__/    # 自動生成物
*.log           # ログ
.DS_Store       # OS が作るゴミファイル
```

> `.env` を `.gitignore` に入れ忘れたままコミットすると、後から消しても履歴には残ってしまいます。**最初に `.gitignore` を用意する** のが安全です。

## 番外編：Fork（公式リポジトリに貢献する）

オープンソースや大会の **公式リポジトリ** に変更を取り込んでもらいたいときは、直接は送れない（権限がない）ため、**Fork** という方法を使います。

* **Fork（フォーク）** … 公式リポジトリを **自分のアカウントに丸ごと複製** すること。GitHub のページで「**Fork**」ボタンを押すだけで作れます。
* 以降、**自分の複製（Fork）＝ `origin`**、**公式 ＝ `upstream`** という2つの送り先を持ちます。

おおまかな流れは、自分の Fork をクローン → 変更して自分の Fork に push → GitHub 上で公式へ **プルリクエスト（PR）** を出す、というものです。PR がレビューされ承認されると、変更が公式に取り込まれます。

> 詳細な手順（作業ブランチの切り方、公式の更新の取り込み方など）は、実際に貢献する段階で必要になります。ここでは「公式へは Fork → PR で貢献する」という流れだけ押さえておけば十分です。

---

## 用語まとめ

| 用語 | 意味 |
|---|---|
| リポジトリ | 履歴つきのプロジェクト一式。手元のものがローカル、GitHub 上のものがリモート |
| クローン（clone） | リモートのリポジトリを丸ごと手元に複製する |
| ステージ（add） | 変更を「次に記録する分」として仮置きする |
| コミット（commit） | 変更をひと区切りとして記録する（1つの記録） |
| プッシュ（push） | 手元の記録を GitHub へ送る |
| プル（pull） | GitHub の最新を手元へ取り込む |
| リモート（remote） | 送り先の GitHub リポジトリ。既定の呼び名が `origin` |
| ブランチ（branch） | 履歴の枝分かれ。`main` を基点に作業用の枝を作って進める |
| Fork | 公式リポジトリを自分のアカウントに複製すること（`upstream`＝公式） |
| プルリクエスト（PR） | 自分の変更を公式に取り込んでもらう依頼 |
