## 仮想環境とは

Pythonの **仮想環境** とは、ひとことで言えば、**プロジェクトごとに独立したPythonの作業場所を用意する仕組み** です。
なぜそれが役に立つのかは、仮想環境を使わないと何が起きるかを考えてみると分かりやすくなります。

### 仮想環境がないと、何が起きるか

Pythonでは `pip install` で手軽にライブラリを追加できます。
ただし何も設定しなければ、追加したライブラリはすべて「パソコン全体で共有される一つのPython」にまとめて入っていきます。
扱うプロジェクトが一つだけのうちは、これで特に困ることはありません。

問題が見えてくるのは、複数のプロジェクトを並行して扱うようになってからです。たとえば——

* プロジェクトAは `langchain` の 0.3 というバージョンを前提に書かれているのに、プロジェクトBは 0.2 でないと動かない。両方を同じ場所に入れると、どちらかが必ず動かなくなる
* 動作確認のためにたくさんのライブラリを入れた結果、どれが必要でどれが不要か分からなくなり、不具合が起きても元の状態に戻せない

どちらも、複数のプロジェクトが **同じ場所を共有している** ことから生まれる問題です。

### 仮想環境が解決してくれること

仮想環境は、この「置き場所」をプロジェクトごとに分けてくれます。
イメージとしては、プロジェクトごとに専用の作業部屋を一つずつ用意するようなものです。
部屋の中でライブラリを入れ替えても、隣の部屋やパソコン全体には影響しません。

その結果、

* あるプロジェクトでライブラリを更新・削除しても、ほかのプロジェクトは影響を受けない
* 環境が壊れても、その部屋ごと作り直せば元通りになる（システム全体のPythonには手を触れずに済む）

といった安心感が得られます。これが、仮想環境を使う最大の利点です。

## 本プロジェクトで使うもの

`aiwolf-nlp-agent-llm` では **[uv](https://docs.astral.sh/uv/)** というツールを使うのが標準です。
`uv` は仮想環境の作成と依存関係のインストールをまとめて行ってくれるので、仮想環境の作成・有効化・パッケージ管理を自分で意識する必要がほぼありません。
前のセクションで挙げた「作業部屋を用意する」「壊れても作り直す」といった操作を、`uv` がほぼ自動で引き受けてくれる、と考えると分かりやすいでしょう。

詳しい導入は [開発環境の準備](../preparation/setup.md) と [リポジトリのクローン](../preparation/clone_repo.md) をご覧ください。

## Pythonの仮想環境ツール比較

参考までに、主な仮想環境ツールの特徴を表にまとめます。本大会では `uv` が推奨ですが、他のツールでも開発できます。

| ツール | 特徴 | ロック/再現性 | 速度 | 向いている場面 |
|---|---|---|---|---|
| **uv** | Rust製の高速ツール。`pyproject.toml` をベースに依存解決 | `uv.lock` | **速い** | 本プロジェクト推奨。素早い環境構築 |
| **venv（標準）** | Python標準の軽量仮想環境 | `requirements.txt` | 普通 | シンプルに使いたいとき |
| **poetry** | 依存解決と配布まで統合 | `poetry.lock` | 中 | ライブラリ開発・厳密な管理 |
| **conda** | Python以外も含めた環境管理 | `environment.yml` | 中 | GPU・科学計算系 |
| **pipenv** | pipとvenvを統合 | `Pipfile.lock` | 中 | Webアプリ開発 |

## venv を使うときの最低限の操作

何らかの理由で `uv` ではなく `venv` を使う場合、以下の操作だけ覚えておけば困りません。

### 作成

```bash
python3 -m venv .venv
```

### 有効化

```bash
source .venv/bin/activate
```

有効化に成功すると、プロンプトの先頭に `(.venv)` と表示されます。
これが、いま「どの作業部屋に入っているか」の目印になります。

### 無効化

```bash
deactivate
```

### `.gitignore` に追加

```text
.venv/
```

仮想環境のフォルダは **コミットしない** のが原則です。
（環境は手元で作り直せるものなので、リポジトリに含める必要がありません）

---

## まとめ

* 仮想環境は、プロジェクトごとにPython環境を分けて、ライブラリの衝突や「戻せない壊れ方」を防ぐための仕組み
* 本プロジェクトでは **`uv`** を使うのが推奨
* どうしても `uv` が使えない環境では `venv` でもOK
