このページでは、人狼知能大会で実際に対戦するときの **全体の流れ** と、予選・本戦に共通する **設定・考え方** をまとめます。

具体的な手順は以下の2ページに分かれています。実際の大会期間中は、自分が今どちらのフェーズにいるかに合わせて読み分けてください。

* **[予選編](./qualifying.md)**：自己対戦して、ログを提出するフェーズ
* **[本戦編](./main_round.md)**：他チームのエージェントと実際に対戦するフェーズ

> ここで紹介する **接続先URL・トークン・スケジュール・大会設定ファイル** は、大会ごとに運営から Slack 等を通じて配布されます。本マニュアルではテンプレートと考え方のみを示し、具体値は載せません。最新の案内は **必ず運営の連絡** を確認してください。

---

## 1. 大会の2段階構成

人狼知能大会は、大きく **2つのフェーズ** で構成されます。

| フェーズ | 内容 | 目的 |
|---|---|---|
| **予選** | 自分のエージェント同士で **自己対戦**。出力ログを運営に提出 | 動作確認と、本戦参加資格の確定 |
| **本戦** | 他チームのエージェントと **実際に対戦**。運営サーバが対戦をマッチング | 大会の順位・受賞対象を決める |

> **重要**：予選の提出が **本戦開始時刻までに確認できない** 場合、自動的に辞退とみなされる運用が一般的です。
> 予選のログ提出は **遅らせず早めに**、所定の方法で行ってください（[予選編](./qualifying.md) を参照）。

---

## 2. トラック（村）について

大会では、参加人数や会話方式によっていくつかの **トラック** が用意されます。
**開催されるトラックの種類は大会ごとに変わります**ので、最新の案内を確認してください。

これまでの大会で実施された例：

| トラック例 | 人数 | Whisper（人狼の囁き） | 会話方式 |
|---|---|---|---|
| 5人村 | 5人 | なし（人狼1人） | ターン制 |
| 9人村 | 9人 | あり（人狼2人以上） | ターン制 |
| 13人村 | 13人 | あり | ターン制 |
| いつでも発話トラック | 5〜13人 | 設定次第 | フリーフォーム（[グループチャット方式](../overview/system_mechanism.md#会話の方式ターン制とフリーフォーム)） |

複数のトラックに参加することもできます。その場合、**それぞれのトラック用に別のエージェントプロセスを起動して、別のポートで待ち受け** する必要があります。

> **ポイント**：トラックごとに `agent.num` の値が違う（5人村なら5、9人村なら9）ので、`config.yml` も **トラックごとに別のファイル** を用意するのがおすすめです。

## 3. 共通の試合設定

過去の大会では、各トラックで以下のような共通設定が採用されました（**大会ごとに変わる可能性があります**。あくまで参考値です。**正確な値はその大会の公式ページ・運営連絡を必ずご確認ください**）。

| 項目 | 値の例 | 意味 |
|---|---|---|
| `talk.max_count.per_agent` | 4 | 1人あたり1日に発言できる最大回数 |
| `talk.max_skip` | 0 | スキップ不可（必ず発言する必要がある） |
| `talk.max_length.base_length` | **日本語：50 / 英語：125** | 1発言の基本文字数 |
| `talk.max_length.mention_length` | **日本語：50 / 英語：125** | `@名前` メンション分の追加文字数 |
| 試合数 | 5回程度 | 1回の接続で連続実行される試合の本数（`matching.game_count`） |

> **言語トラックで値が変わります**：上記の通り、`base_length` / `mention_length` は **日本語トラックと英語トラックで異なる値** が採用されています。
> 英語のほうが情報量あたりの文字数を要する傾向への配慮として、英語トラックでは大きめの値が設定されています。

これらの値は **サーバ側の設定ファイル** で決まっており、参加者が個別に変えるものではありません。
ただし、**プロンプトを書くときに「あと何文字書けるか」を意識する** 上で頭に入れておくと有利です。

> **`@名前` のメンション機能**：発言の中で **最初に出てきた `@プレイヤー名`**（例：`@ミナト`）が検出されると、その名前以降の部分に対して `mention_length` ぶんの追加枠が許可されます。
> ※ 複数メンションを書いても拡張されるのは **最初の1か所のみ** です。詳しくは [大会のルールとゲームの流れ](../overview/aiwolf_logic.md) を参照してください。

## 4. 全フェーズに共通する `config.yml` の設定

予選・本戦のどちらでも、`config/config.yml` の `web_socket` セクションを **運営配布の値に書き換える** のが基本です。

```yaml
web_socket:
  url: ws://example.com:8000/ws   # 運営から配布された接続先URL（トラックごとに異なる）
  token:                          # 大会によっては不要。配布された場合のみ設定
  auto_reconnect: true            # ★必ず true にする（後述）

agent:
  num: 5                          # トラックの人数に合わせる（5 / 9 / 13 など）
  team: kanolab                   # 参加登録時のチーム名と一致させる
  kill_on_timeout: true
```

**特に重要なポイント：**

* **`auto_reconnect: true`** にする
  * 1回の接続で連続5試合などの形で進行するため、`false` だと **試合終了後に切断** したきり次の試合に参加できません
  * 接続トラブル時の自動再接続にも必須
* **`agent.num`** は **トラックの人数と一致** させる
  * 5人村に `num: 9` で繋いでも正常に動作しません
* **`agent.team`** は **参加登録したチーム名** と一致させる
  * マッチングや成績集計の識別子に使われます
* **接続先 URL（`web_socket.url`）はトラックごとに異なります**
  * 5人村と9人村では別ポート／別パスのことが多く、必ず運営の案内通りに設定してください

## 5. 起動方法

通常のローカル対戦と同じコマンドで起動できます。

```bash
cd ~/aiwolfdial/aiwolf-nlp-agent-llm
uv run python src/main.py
```

複数のトラックに同時参加する場合は、**ターミナルを複数開いて、それぞれで別の `config.yml` を読ませて起動** します。
たとえば次のように、トラックごとにフォルダを分けるとシンプルです。

```bash
# ターミナル1：5人村用
cd ~/aiwolfdial/aiwolf-nlp-agent-llm
cp config/config_5p.yml config/config.yml
uv run python src/main.py

# ターミナル2：9人村用（別ターミナル）
cd ~/aiwolfdial/aiwolf-nlp-agent-llm
cp config/config_9p.yml config/config.yml
uv run python src/main.py
```

> **注意**：同じディレクトリの `config/config.yml` を上書きする運用だと、起動順を間違えるリスクがあります。
> 慣れてきたら `python src/main.py --config config/config_5p.yml` のように **コマンドライン引数で切り替える改造** も検討してください（要コード追加）。

## 6. 期間中の心がけ

### API レート制限・残量

連続5試合 × 複数トラックを長時間動かすと、**LLM API のリクエスト上限・課金額が大きくなります**。

* 大会前に **API キーの残量を確認**
* 必要であれば **課金プランの引き上げ** を済ませておく
* OpenAI / Google それぞれで **1分あたりリクエスト数（RPM）** に注意

### エージェントの停止と再起動

本戦中でも、エージェントを **一時停止して `config.yml` を直し、再起動する** ことは可能です。
ただし、停止中はその試合に参加できないので、**実施するなら試合と試合の間** が望ましいです。

### 連絡手段

接続トラブルや質問は、**運営から指定された Slack チャンネル等で速やかに連絡** してください。
本戦のような時間制約のあるフェーズでは、**エラーログの先頭数十行を添えると即対応してもらえる** ことが多いです。
